### D1 — Locked: the **runner** supplies the weights; the door keeps ONE source

R4 shows the door already weights off `body.dimension_weights`, read by all four branches. Two ways to close
the column's loop, and only one is safe:

1. **(REJECTED) The door resolves `run_id` → `eval_runs.dimension_weights`.** The door would then have **two**
   sources for one value (`body.dimension_weights` and the column) and would need a precedence rule between
   them. That is a **priority fallthrough** — the anti-pattern CLAUDE.md names explicitly ("No priority-based
   fallthrough"), and the precise shape of E-4's D4 hazard, where `run_eval` dispatching by priority let
   `MODE=webhook` fall through to the reactive tail and deliver real side effects under a plausible PASS.
2. **(LOCKED) The eval-runner reads the run once, and passes `dimension_weights` explicitly in the score body.**
   One source at the door (`body.dimension_weights` — unchanged, all four branches keep working verbatim), one
   reader of the column (the runner), no precedence rule to get wrong. The runner **already** fetches its run's
   context and **already** owns the per-item `passed` verdict, so it is where a per-run policy belongs.

The same argument fixes the threshold: the runner reads `run.pass_threshold` **once** at startup (T004) and it
becomes the **only** number both the per-item `passed` verdict and the gate use (D2).

### D2 — Locked: ONE threshold, resolved once, read by every consumer

R5's four copies are not four bugs; they are **one** design flaw — a policy value with no owner, re-declared
wherever it was needed. Patching `run.pass_threshold` into the gate alone (plan T1) fixes **one instance** and
leaves the product **lying** in the other three. CLAUDE.md: *"Does this fix the class of problem, or just this
instance? If just this instance, go deeper."*

**Fix the design — one resolution, three consumers, zero re-declarations:**

- **Owner:** `eval_runs.pass_threshold` (the column), defaulted at **write** time (T001) from the global
  `EVAL_PASS_THRESHOLD` when the caller does not specify one. **After T001 the column is NEVER NULL on a new
  row** — so downstream has a value to read and needs no fallback of its own.
- **Consumer 1 — the gate** (`eval_runner.py:523`/`:540`): reads `run.pass_threshold` (T002).
- **Consumer 2 — the per-item `passed` verdict** (`eval-runner/main.py:563/867/1098/1306/1416`): reads the
  threshold **resolved from the run**, not `_JUDGE_PASS_THRESHOLD` (T004).
- **Consumer 3 — the UI verdict + colour band** (`EvalResultsPage.tsx:194`/`:51`): reads `run.pass_threshold`
  off the API response (already on `EvalRunResponse`, `schemas.py:1523` — the TS type just omits it) (T006).

`EVAL_PASS_THRESHOLD` survives as the **platform default at the single write site** (T001) and nowhere else.
`JUDGE_PASS_THRESHOLD` (the runner's env) survives **only** as the fallback for a run whose column is NULL —
i.e. rows written **before** T001 — and is asserted to have **no other reader** (`T-S80-000c`). One number,
one owner, three readers; a per-run threshold that reaches the gate but not the UI is the bug, not the goal.
